# 踩坑记录 —— AHT20 + ILI9341 LCD 移植

## 坑1：LCD 源文件未加入 Keil 工程

**现象**：编译通过，链接失败，报 `Undefined symbol LCD_Init`、`LCD_Fill`、`LCD_String`。

**原因**：只把 `.c/.h` 复制到了 `Core/Src/` 和 `Core/Inc/`，但 `.uvprojx` 工程文件里没有注册这个源文件。

**解决**：在 `.uvprojx` 的 `<Groups>` → `Application/User/Core` 组中添加：
```xml
<File>
  <FileName>bsp_LCD_ILI9341.c</FileName>
  <FileType>1</FileType>
  <FilePath>../Core/Src/bsp_LCD_ILI9341.c</FilePath>
</File>
```

## 坑2：PB6/PB7 引脚冲突 —— LCD 与 I2C 互踩

### 硬件背景

| 功能 | 引脚 |
|------|------|
| LCD 数据 D6、D7 | PB6、PB7 |
| AHT20 I2C SCL、SDA | PB6、PB7 |

板子（苏宙嵌孝 STM32-F103RCT6 V5.4.1）的硬件上，PB6 和 PB7 同时接到了 LCD 16 位并行数据线和板载 I2C 总线（外接 10K 上拉电阻）。**两个外设共用同两条引脚。**

### 现象

| 当... | 会发生什么 |
|-------|-----------|
| LCD 写数据（`GPIOB->ODR = data`） | PB6/PB7 被强制设为数据位，I2C 总线被干扰，AHT20 读到错误数据 |
| I2C 通信后 PB7 处于输入模式 | LCD 写 D7 时引脚不驱动，屏幕花屏或局部无显示 |

### 为什么硬件 I2C（HAL 库）不能用

`MX_I2C1_Init()` 会把 PB6/PB7 配置为复用功能（AF），而 `LCD_Init()` 里 `GPIOB->CRL = 0x11111111` 会把 PB0~PB7 **全部覆盖为通用推挽输出**。两个初始化互相覆盖，时序不可控，数据必然出错。

### 解决：软件模拟 I2C + 用完即恢复

```c
// I2C 通信结束后的关键一步：
static void sda_restore_for_lcd(void)
{
    GPIOB->CRL &= ~(0xFU << (7 * 4));
    GPIOB->CRL |=  (0x1U << (7 * 4));  // PB7 恢复为推挽输出
    GPIOB->BSRR = GPIO_PIN_7;
}
```

核心原则：
1. PB7 在 I2C 需要驱动低电平时切为开漏输出，需要读/释放时切为输入上拉
2. **每次 I2C 通信结束后**立即调用 `sda_restore_for_lcd()` 恢复推挽输出
3. PB6(SCL) 始终推挽输出，不需要恢复

## 坑3：font.h 中文字符编码

### 现象

```
error: #144: a value of type "char [7]" cannot be used to initialize
       an entity of type "signed char [2]"
```

### 原因

`font.h` 里 `z_GB_16[]` 结构体的 `Index[2]` 字段是中文字符串字面量，原本按 GB2312（2 字节/字）保存。文件流转过程中编码变成了 UTF-8（3 字节/字），ARMCC V5 无法将 3 字节的 UTF-8 码点塞进 `signed char[2]`。

### 解决

不要用字符串字面量，直接写原始的 2 字节 hex 值：

```c
// ❌ 错误：编码不确定
"温", 0x04,0x44,...

// ✅ 正确：用原始 GB2312 hex 字节
0xCE,0xC2, 0x04,0x44,...
```

| 汉字 | GB2312 码 |
|------|----------|
| 温 | `0xCE, 0xC2` |
| 嵌 | `0xC7, 0xB6` |
| 入 | `0xC8, 0xEB` |
| 式 | `0xCA, 0xBD` |

## 坑4：ARMCC V5 移位溢出

### 现象

```
warning: #61-D: integer operation result is out of range
warning: #68-D: integer conversion resulted in a change of sign
```

### 原因

`0xF << 28` 的结果 `0xF0000000` 超过了 `int32` 的范围（`0x7FFFFFFF`），因为字面量 `0xF` 默认是 `signed int`。

### 解决

所有参与移位的常量加 `U` 后缀：
```c
// ❌ 警告
GPIOB->CRL &= ~(0xF << (7 * 4));

// ✅ 正确
GPIOB->CRL &= ~(0xFU << (7 * 4));
```

## 经验总结

1. **复制 `.c` 文件到项目目录 ≠ 加入工程**，Keil 需要显式在 `.uvprojx` 中注册。
2. **多外设共用引脚时**优先画一张引脚占用表，确认有无冲突再动手编码。
3. **含中文的源文件**跨平台/跨工具流转时编码可能变，初始化数据用 hex 值最稳。
4. ARMCC V5 对整数字面量类型严格，移位运算记得加 `U`。
