## 零、最终目标

STM32 读 AHT20 温湿度 → 串口发给 ESP8266 → ESP8266 经 WiFi → 网络调试助手（PC TCP Server）
→ PC 收到 JSON 数据

> 当前阶段：**不接真云端**，先用网络调试助手当 TCP Server 验证整条链路。
> 之后可以无缝切换到 Aliyun / OneNET 物联网平台。

## 一、最终改动清单

### 1.1 硬件确认（最关键）

| 实物 | 实际位置 |
|---|---|
| **板载 ESP8266 TX** | **PC11**（丝印"UART4_RX"） |
| **板载 ESP8266 RX** | **PC10**（丝印"UART4_TX"） |
| **跳线帽位置** | PC10/PC11 旁边的跳线帽 → 短接 **ESP8266** 一侧（不要短 RS-485） |

> 板子丝印"UART4_TX/RX"是误导：RCT6 没有 UART4/5，实际是 **USART3 部分重映射到 PC10/PC11**。

### 1.2 代码改动总览

| 文件 | 改动 | 原因 |
|---|---|---|
| `Inc/pins.h` | USART2 宏 → **USART3** 宏（PC10/PC11） | ESP8266 实际接 PC10/PC11 |
| `Core/Inc/main.h` | `huart2` → `huart3` | 同上 |
| `Core/Inc/stm32f1xx_it.h` | `USART2_IRQHandler` → `USART3_IRQHandler` | 同上 |
| `Core/Src/main.c` | `MX_USART2_UART_Init` → `MX_USART3_UART_Init`，全局 `huart2` → `huart3`，删 `#include "mqtt.h"` | 切换 UART；mqtt 暂不用 |
| `Core/Src/stm32f1xx_it.c` | `USART2_IRQHandler` → `USART3_IRQHandler`，`huart2` → `huart3` | 切换中断 |
| `Core/Src/stm32f1xx_hal_msp.c` | USART3 MspInit 加 `__HAL_AFIO_REMAP_USART3_PARTIAL()` 和 `__HAL_RCC_AFIO_CLK_ENABLE()` | **关键**：把 USART3 从默认的 PB10/PB11 重映射到 PC10/PC11 |
| `Drivers/BSP/ESP8266/esp8266.c` | `ESP8266_Reset()` 跳过硬件 RST 拉低，改成直接发 `AT` 测试 | 板载 ESP8266 RST 引脚未知，跳过最稳 |
| `Drivers/BSP/ESP8266/aliyun_iot.c` | 改成"调试版"：`Aliyun_PropertyPost` 走 ESP8266 直推 TCP 到网络调试助手 | 调试阶段不上云，先验证链路 |
| `MDK-ARM/ahd20.uvprojx` | 清理 AHT20 组里指向已删除文件的死引用（esp8266.c / mqtt.c / aliyun_iot.c），ESP8266 组删 mqtt / sha1 引用 | 防止重复定义 + 链接错误 |

### 1.3 删除文件

| 路径 | 原因 |
|---|---|
| `Core/Src/esp8266.c` / `Core/Inc/esp8266.h` | 跟 `Drivers/BSP/ESP8266/` 里的旧版重复 |
| `Core/Src/mqtt.c` / `Core/Inc/mqtt.h` | 跟 `Drivers/BSP/ESP8266/mqtt.c` 重复 |
| `Core/Src/aliyun_iot.c` / `Core/Inc/aliyun_iot.h` | 跟 `Drivers/BSP/ESP8266/aliyun_iot.c` 重复 |
| `Drivers/BSP/ESP8266/mqtt.c` / `.h` | 调试阶段不用真 MQTT |
| `Drivers/BSP/ESP8266/sha1.c` / `.h` | 调试阶段不用 HMAC-SHA1（真 Aliyun 才需要） |
| `Drivers/BSP/ESP8266/aliyun_iot.c`（原版）| 改成"调试版"覆盖（已备份逻辑在脑子里） |

## 二、踩坑顺序（按出现时间）

### 坑 1：以为 ESP8266 接 USART2（PA2/PA3）

**现象**：所有改动做完，AT 不响应，串口打 `[ESP] Reset failed`

**真相**：
- 板子丝印 PC10/PC11 标"UART4_TX/RX"——是 **USART3 部分重映射**，不是 USART2
- 我们一开始在 `pins.h` 写的 `USART2 (PA2/PA3)` 完全接错

**修复**：
- `pins.h` 改 USART3 (PC10/PC11)
- `main.c` 改 `MX_USART2_UART_Init` → `MX_USART3_UART_Init`
- `stm32f1xx_it.c` 改 `USART2_IRQHandler` → `USART3_IRQHandler`
- 中断里用 `&huart3`

### 坑 2：USART3 默认不在 PC10/PC11

**现象**：改成 USART3 后 AT 还是不响应

**真相**：
- STM32F103RCT6 的 USART3 **默认**在 **PB10/PB11**
- PC10/PC11 是**部分重映射**，需要 `__HAL_AFIO_REMAP_USART3_PARTIAL()` 开启
- 板子设计默认就开了重映射，但 HAL 库代码里**必须显式调用**
- 不然 USART3 跑在 PB10/PB11，PC10/PC11 配的 GPIO 是空的

**修复**（`stm32f1xx_hal_msp.c`）：
```c
__HAL_RCC_AFIO_CLK_ENABLE();
__HAL_AFIO_REMAP_USART3_PARTIAL();  // ← 关键
```

### 坑 3：esp8266.c 的 Reset 函数依赖硬件 RST

**现象**：AT 通了（能看到 RST 在跑），但 `[ESP] Reset failed`

**真相**：
- 老的 `esp8266.c` 的 `ESP8266_Reset()` 用 `ESP_RST_LOW()/HIGH()` 拉低 PC4 100ms
- 然后发空字符串等"ready"
- 板载 ESP8266 的 **RST 引脚可能不是 PC4**（或者 ESP8266 一直在跑、不会重发"ready"）
- 等等还有 `s == ESP8266_ERROR` 时不进入 AT 兜底分支

**修复**（`esp8266.c` 的 `ESP8266_Reset`）：
- 跳过硬件 RST 拉低（注释掉 `ESP_RST_LOW()/HIGH()`）
- 直接 `send_at_cmd("AT", "OK", 3000)`，重试一次

### 坑 4：Keil 工程树引用了已删除的文件

**现象**：编译报 `cannot open source input file "Drivers/BSP/ESP8266/mqtt.c"`

**真相**：
- 我删了 `mqtt.c / mqtt.h / sha1.c / sha1.h` 这 4 个文件
- 但 `ahd20.uvprojx` 里还有它们的引用
- 编译时找不到文件

**修复**：手动改 `.uvprojx`，删掉这些引用（AHT20 组的旧引用 + ESP8266 组的 mqtt/sha1 引用）

### 坑 5：main.c 包含不存在的 mqtt.h

**现象**：编译报 `cannot open source input file "mqtt.h"`

**真相**：`main.c` 写了 `#include "mqtt.h"`，但 mqtt.h 已被删

**修复**：`main.c` 删掉 `#include "mqtt.h"`

### 坑 6：Keil 编译缓存了旧 .o 文件

**现象**：改了 `aliyun_iot.c` 的 IP/端口，但烧录后 log 还是旧的 `192.168.1.100:8888`

**真相**：
- `aliyun_iot.o` 没被删，Keil 直接 link 旧 .o
- `Clean Targets` 不一定把 .o 删干净

**修复**：
- 删 `MDK-ARM/Objects/aliyun_iot.o` 强制重编
- 或者 `Project → Clean Targets` 后再 F7

### 坑 7：PC 的真实 WiFi IP 不是你以为的那个

**现象**：改了 `TCP_SERVER_IP = 192.168.1.100`，但 ESP8266 连不上

**真相**：
- `ipconfig` 显示 **2 个 IPv4**：
  - `192.168.56.1` → **VMware 虚拟网卡**（不是真 WiFi）
  - `10.117.237.22` → **手机热点 WiFi**（这个才是真 IP）
- 一开始错用了 VMware 那个

**修复**：`ipconfig` 找带手机热点的那个，填到 `aliyun_iot.c`

### 坑 8：网络调试助手没正确配置

**现象**：TCP 还是连不上，PC 防火墙已关、IP/端口都对

**真相**：
- 调试助手（NetAssist）配置错了：
  - **本地 IP** 选了 `192.168.56.1`（VMware 那张网卡）
  - **服务器没启动**（红色按钮显示"关闭"）
- 就算 ESP8266 发 TCP 过来，调试助手也没在那监听

**修复**：
- 改调试助手的本地 IP → `10.117.237.22` 或 `0.0.0.0`
- 点红色按钮启动服务器

## 三、关键概念速查

### 3.1 USART3 重映射

| USART3 引脚 | 默认 | 部分重映射 | 完全重映射 |
|---|---|---|---|
| TX | PB10 | **PC10** | PD8 |
| RX | PB11 | **PC11** | PD9 |

> 64 脚 LQFP 不支持完全重映射，只能用部分重映射。

### 3.2 ESP8266 AT 调试流程

```
1. AT                     → OK              (测试模块响应)
2. ATE0                   → OK              (关回显，看着清爽)
3. AT+CWMODE=1            → OK              (Station 模式，连接外部 WiFi)
4. AT+CWJAP="SSID","PWD"  → OK              (连 WiFi，最长 15s)
5. AT+CIPMUX=0            → OK              (单连接模式)
6. AT+CIPSTART="TCP","IP",port → OK         (建 TCP 连接)
7. AT+CIPSEND=len         → > (提示符)      (准备发数据)
   ... raw data ...
8. (收到) SEND OK                          (发送完成)
```

### 3.3 PC IP 查询

```powershell
ipconfig
```

找 `IPv4 Address`：
- 真 WiFi 卡的 IP（手机热点 / 路由器）
- **不要**选 VMware / VirtualBox / Hyper-V 的虚拟网卡 IP

## 七、避坑 checklist（下次别再踩）

1. **板载外设引脚** → 查 `board_pinout.md`，别凭印象
2. **USART 引脚有重映射** → HAL 代码里要显式开 `__HAL_AFIO_REMAP_*`
3. **改 .c 文件 → Clean Targets → F7** → 不然可能 link 旧 .o
4. **ipconfig 看 IPv4** → 选手机热点/路由器的网卡，别选 VMware 虚拟网卡
5. **网络调试助手** → 本地 IP 不要选 127.0.0.1，必须 0.0.0.0 或本机 IP
6. **网络调试助手** → 红色按钮要点开，状态显示"已监听"才有效
7. **手机热点** → 优先 2.4GHz（ESP8266 不支持 5GHz）
8. **PC 和 ESP8266 必须在同一 WiFi** → 同一手机热点或同一路由器
